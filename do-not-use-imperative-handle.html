React - Não useImperativeHandle

January 1, 2023 · 11 min read
Victor Heringer
Developer

Recentemente scrollando aleatóriamente pelo Linkedin por alguns minutos sem ver nada de interessante, eu finalmente acabei me deparando com uma postagem que chamou minha atenção: era uma sugestão de utilização do useImperativeHandle para gerenciar abertura e fechamento de modals. Segui lendo a explicação para entender como esse hook resolvia esse tipo de problema, e ao terminar de ler parei para pensar um pouco no que estava sendo proposto ali.

O fato é que a primeira coisa que veio a minha mente foram as sábias palavras da doc do React que deixam explicito que:

Como sempre, na maioria dos casos, seria bom evitar um código imperativo usando refs.

Como um hook tão específico poderia resolver um problema tão trivial como lidar com um modal? Me peguei alguns minutos pensando sobre, lendo os comentários da postagem relendo a documentação e cheguei na conclusão mais óbvia possível, não pode.

Continuei pensando sobre, relendo os comentários e mais uma coisa também me chamou a atenção, a quantidade de pessoas que achou uma ótima solução. Parando para refletir mais um pouco, a conclusão que cheguei é que grande parte das pessoas que estão entrando agora no mercado de tecnologia, acabando seguindo um caminho de aprendizado que as colocam mais rápido no mercado de trabalho (o que vale ressaltar, não tem problema nenhum), porém em contrapartida, alguns conceitos da base acabam ficando para segundo plano. Devido a isso decidi deixar registrado aqui um pouco mais sobre o useImperativeHandle para ajudar quem quiser entender um pouco mais, e para minhas próprias referências futuras quando eu não lembrar exatamente mais pra que isso serve.

O que é o useImperativeHandle ?​

Esse hook tem uma função extramente simples e bem definida, ele permite customizar uma ref, onde essa ref em React faz referência (piadas a parte) a um elemento do DOM.

Em javascript puro, isso seria algo como usar um getElementById para obter um elemento do DOM, porém no caso das refs o que fazemos é definir uma variável e atribui-la a esse elemento representado através da sintaxe do jsx.

const ref = document.getElementById("meuInput");

Seria entre algumas aspas equivalente à:

const ref = useRef(null);

return <input ref={ref}>

Em ambos os casos você poderia usar a função de focus() desse input, tomando cuidado com o fato de que uma referência no React deve ser usada a partir do deu valor atual, utilizando o .current.

Nesse momento você talvez esteja se perguntando então o porque de não usar o getElementById no lugar do useRef, mas eu vou te parar aqui mesmo para a gente evitar que seja feito uma tangente gigante sobre o assunto e simplificar as coisas dizendo que o useRef ou createRef é o jeito "React" de lidar com elementos do DOM.

Continuando, essas referências então são responsável por permitirem que uma coisa muita importante seja feia: através delas o componente pai pode chamar funções de um componente filho. E isso é totalmente contra intuitivo em relação ao "fluxo natural" do React onde o pai define funções e props e repassam para que os filhos possam usa-lás. Ou seja, o fluxo de dados no React é sempre "de cima para baixo", mas as refs permitem que a um componente filho seja modificado fora desse fluxo "natural" do React.

É aqui que entra um ponto importantíssimo, o React utiliza um paradigma declarativo onde os componentes e hooks devem saber como reagir a uma mudança de estado, ou seja você, altera um estado, causa um novo ciclo de render e todos os componentes respondem aquela mudança.

E em contrapartida, as refs permitem que um componente seja modificado imperativamente (declarando explicitamente como algo deve acontecer) o que é um paradigma diferente do estabelecido na lib. Logo quando você utiliza:

ref.current.focus();

Você está dizendo imperativamente para que seu elemento seja focado, e não fazendo com o que ele reage a uma mudança de estado.

Eu acredito que pode ser um pouco confuso a princípio essa diferenciação entre um código imperativo e declarativo, principalmente quando o exemplo em questão utiliza apenas uma linha código, mas o que eu eu gostaria que você tenha em mente a princípio é que se você precisa dizer o passo a passo de algo, esse algo é imperativo.

Eu pretendo fazer em algum momento um post aprofundando sobre o assunto de programação imperativa vs declarativa, e faço um link aqui para ajudar nessa explicação.

Então sintetizando, usar referências está associado a escrever um código imperativo e não é esse o paradigma que o React prega. Com isso entendido, fica fácil perceber porque o useImperativeHandle só deve ser usados em casos estritamente necessários pois sua única função, como dito anteriormente, é permitir com que você altere o valor de uma referência. Então, vamos para um exemplo:

Vamos supor que você precise modificar o comportamento natural de uma função de focus de um input para que toda vez que esse foco seja feito, a cor de fundo dele mude. Você então poderia se aproveitar do useImperativeHandle para atingir esse objetivo:

function ComponenteFilho(props, ref) {
  const inputRef = useRef();
  const [color, setColor] = useState("#fff");

  useImperativeHandle(ref, () => ({
    focusWithColor: (nextColor: string) => {
      inputRef.current.focus();
      setColor(nextColor);
    },
  }));

  return <input style={{ backgroundColor: color }} ref={inputRef} />;
}

ComponenteFilho = forwardRef(ComponenteFilho);

function ComponentePai() {
  const ref = useRef(null);

  return (
    <>
      <button onClick={() => ref.current.focusWithColor('#f00')}>
      <ComponenteFilho ref={ref} />
    </>
  );
}

Este exemplo mostra como podemos utilizar este hook, porém esse é apenas um exemplo didático do funcionamento do mesmo, e não um caso de uso apropriado para aplica-ló.

Porém, olhando o código acima pode parecer extremamente tentador usar o useImperativeHandle para lidar com modals, afinal, se você consegue mudar um estado do filho, sem declara-lo no pai, você não precisaria ter o estado de isVisible e as funções de abrir e fechar o modal em cada um dos componentes pais que os utilizam, bastava por toda essa lógica no componente de modal (que é o filho), modificar a ref para adicionar as funções de abrir e fechar o modal, e pronto os pais só precisaria passar uma ref e poderiam chamar os métodos a hora que quiser, Então vamos implementar:

function ComponenteFilhoModal(props, ref) {
  const inputRef = useRef();
  const [isVisible, setIsVisible] = useState(false);

  useImperativeHandle(ref, () => ({
    open: () => {
      setIsVisible(true);
    },
    close: () => {
      setIsVisible(false);
    }
  }));

  return isVisible <p>Hello modal</p>;
}

ComponenteFilhoModal = forwardRef(ComponenteFilhoModal);

function ComponentePai() {
  const ref = useRef(null);

  return (
    <>
      <button onClick={() => ref.current.open()}>
      <ComponenteFilhoModal ref={ref} />
    </>
  );
}

Olhando o código, você percebe que a lógica estaria abstraída no filho e parece que você resolveu um problema de duplicação desnecessária de código, afinal, se a lógica estivesse no pai, cada um dos pais que precisa usar um modal, também teria que ter a lógica de abrir e fechar levando a duplicação. Mas o que você na verdade você fez foi criar um outro problema, que é usar o React da maneira oposta que ele foi planejado para ser utilizado.

E como o react foi planejado para ser usado? Bom este problema da implementação e abstração da lógica de abrir e fechar um modal assim como vários outros são conhecidos como problemas de compartilhamento de lógica de estado, e podem ser resolvidos através da feature que o React criou justamente para isso que são, adivinhem só... os hooks! E como podemos ver na doc:

Dois componentes usando o mesmo Hook compartilham estado (state)? Não. Hooks customizados são um mecanismo para reutilizar lógica com estado (como > configurar uma subscrição ou lembrar de um valor atual), mas sempre que você usa um Hook customizado, todo o estado (state) e os efeitos dentro dele são completamente isolados.

Enão os hooks são a forma que o React encontrou para compartilhamento de lógica de estado, e criar um hook como um useModal resolveria seu problema de duplicação de lógica, além de manter a característica daquele fluxo natural de dados onde o pai chamaria os hooks e repassaria o estado de aberto ou fechado para o filho, e pronto problema resolvido!

function useModal() {
  const [isVisible, setIsVisible] = useState(false);

  function open() {
    setIsVisible(true);
  }

  function close() {
    setIsVisible(false);
  }

  return { isVisible, show, hide };
}

function ComponenteFilhoModal({ isVisible }) {
  return isVisible <p>Hello modal</p>;
}

function ComponentePai() {
  const { isVisible, open, close } =  useModal()

  return (
    <>
      <button onClick={() => open()}>
      <ComponenteFilhoModal isVisible={isVisible} />
    </>
  );
}

E se todas essas explicações ainda não foram o suficiente para te mostrar que usar o useImperativeHandle para modais é um "anti-pattern", o React em sua doc nos da um exemplo bem específico:

Evite usar refs para qualquer coisa que possa ser feita de forma declarativa. Por exemplo, ao invés de expôr os métodos open() e close() em um componente Dialog, passe a propriedade isOpen para ele.

Quando devo utilizar o useImperativeHandle ?​

Agora que você entendeu pra que serve o useImperativeHandle e que não deveria utiliza-lo porque ele vai contra os princípios do React você deve estar achando que este hook é um surto coletivo ou pegadinha do time que desenvolve o React, certo?

E não é pra menos, pra toda linha que você lê na doc do react falando em refs, haverão duas embaixo delas dizendo para não usa-las. Porém as refs, assim como useImperativeHandle, são necessárias, porque a implementação dos elementos do DOM não são declarativas, e por isso as vezes temos que quebrar conscientemente algumas boas práticas e padrões na área de tecnologia, pois nem tudo vai funcionar exatamente da maneira exata que a gente precise que funcione, por isso é importantíssimo entender exatamente o que estamos fazendo e motivo pelo qual estamos fazendo.

Eu particularmente precisei usar pouquíssimas vezes esse hook, e uma das vezes que houve necessidade foi por conta de especificidades do react-native e a forma com que funciona as positions nele, e por conta disso, eu precisei que um componente filho, alterasse um outro componente filho, sem poder por essa lógica no componente pai, e compartilha-la entre eles.

Esse é um dos principais motivos pelo qual fica difícil dizer quando esse hook deveria ser usado, são casos de borda que vão variar de projeto pra projeto, negócio para negócio e arquitetura para arquitetura. O importante é você saber o que ele faz e as compensações que ele traz para que no momento em que você precise dele, você consiga tomar a melhor decisão.

Eu terminaria esse post por aqui, mais tem mais um ponto que acho muito relevante para fechar esse assunto.

Por que utilizamos o forwardRef com useImperativeHandle?​

A doc do React simplesmente diz:

O useImperativeHandle deve ser usado com forwardRef

E é isso vamos que vamos? Bom acho que não né, é importante entender porque precisamos dessa função, então porque de fato à usamos?

Bom o forwardRef serve para encaminhar uma referência. Partindo do nosso exemplo, nós precisamos que a ref declarada no ComponentePai seja repassada, para que possamos manda-lá como parâmetro para o useImperativeHandle, pois nós não queremos que a ref declarada no pai, seja uma ref para o ComponenteFilho como aconteceria caso nós não a tivesse repassado utilizando o forwardRef. Ou seja, no fim das contas nós não queremos que o ComponentePai tenha uma referência para o ComponenteFilho, nós queremos que essa referência seja repassada para que possamos criar nela o método focusWithColor, de maneira que essa ref que estamos setando utilizando o useImperativeHandle, tenha acesso a referência do input, que é declarada dentro do próprio filho.

E agora sim, é isso.
